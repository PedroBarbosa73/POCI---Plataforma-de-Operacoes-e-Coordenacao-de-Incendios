# POCI — Supabase Backend & Real-time Sync

**Date:** 2026-03-20
**Status:** Approved

## Overview

Replace the current localStorage/mock-data architecture with a Supabase backend. All state becomes persistent, multi-user, and real-time. The app is deployed to Vercel and shared via a public URL.

## Goals

- Persistent data that survives page refreshes and browser closes
- Real-time sync: changes made by one user appear on all connected clients within ~200ms
- Proper auth: Supabase Auth replaces NextAuth
- Deployable: app runs on Vercel, accessible via URL to anyone with credentials
- Migrated seed data: existing mock incidents, units, zones, alerts etc. pre-loaded into Supabase

## Out of scope

- Role-based access control (beyond authenticated vs. public)
- File/image uploads
- Push notifications
- Fire spread prediction model
- Mobile-specific layout changes

## Database Schema

All tables live in Supabase (PostgreSQL). All have Realtime enabled (`REPLICA IDENTITY FULL`, added to `supabase_realtime` publication).

**ID generation:** All PKs use `gen_random_uuid()` as default except tables where the existing string IDs (e.g. `INC-034`, `BV-034-01`) are meaningful and used in UI/routing — those keep text PKs with client-supplied values. Specifically: `incidents`, `units`, `fire_stations` use text PKs. All other tables use `uuid` PKs generated by Supabase.

## Auth

- **Provider:** Supabase Auth (email + password)
- NextAuth and `next-auth` dependency removed entirely
- `middleware.js` rewritten using `@supabase/ssr` to check session cookie
- Protected routes (require auth): `/comando`, `/meios`, `/radio`, `/alertas`, `/relatorio`, `/demo`
- Public routes (no auth): `/publico`, `/login`
- `useSession` calls replaced with a `useSupabaseUser()` hook wrapping `supabase.auth.getUser()`; all pages that currently import `useSession` from `next-auth/react` must be updated: `/alertas/page.js`, `/radio/page.js`, `/demo/page.js`, and `NavBar.js` (which also imports `signOut`)
- `signOut()` replaced with `supabase.auth.signOut()`
- Login page (`/login`) fully redesigned: replaces the current OAuth-style button UI with an email/password form calling `supabase.auth.signInWithPassword()`
- Users managed via Supabase dashboard — no user management UI in the app
- `app/app/api/auth/` directory deleted
- `SessionProvider` removed from `providers.js` and `layout.js`

## Row Level Security (RLS)

RLS is enabled on all tables:
- `authenticated` role: full SELECT, INSERT, UPDATE, DELETE
- `anon` role: SELECT only on `incidents`, `alerts`, `closures` (used by `/publico`)
- All other tables: no `anon` access

This ensures the `NEXT_PUBLIC_` anon key cannot be used to write data from outside the app.

## Real-time Strategy

`usePociState` is rewritten as the central data hook:

1. **Initial load:** fetch all tables via Supabase client on mount
2. **Subscriptions:** one Supabase Realtime channel per table — INSERT/UPDATE/DELETE events update React state in place
3. **Writes:** all mutations call Supabase directly; the realtime subscription propagates the change to all clients automatically
4. **Optimistic updates:** for latency-sensitive actions (unit status, assignment), update local state immediately then confirm via Supabase. On conflict, last-write-wins — the realtime event corrects the local state

### Scenario state
`scenarioActive` and `currentStep` (demo player state) remain in `localStorage` — they are session-local and do not need to sync across users. `usePociState` keeps this slice in localStorage after migration.

## Seed Migration

A one-time seed script (`app/scripts/seed.js`) reads from `mockData.js` and inserts all records into Supabase via the service role client. Run once after schema creation. After that, `mockData.js` is no longer imported at runtime.

Seeded data includes:
- 4 incidents with their `incident_communications`; field names are transformed from camelCase to snake_case (e.g. `brushRadiusKm` → `brush_radius_km`, `windDeg` → `wind_deg`; `windDeg` is absent from mock data so the column seeds as `null`)
- All fire stations
- All units (including `air_kind` mapped from `airKind` for air units)
- `unit_states` (assignment + status derived from mock distribution)
- Initial zones and closures
- Mock alerts
- Mock radio/digital messages as `op_log` rows
- Op log history seeded entries

## Deployment

1. Supabase project created at supabase.com (free tier, EU region)
2. Schema applied via Supabase SQL editor
3. RLS policies applied
4. Seed script run once: `node app/scripts/seed.js`
5. Vercel project created for `/app` directory
6. Environment variables set in Vercel:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `SUPABASE_SERVICE_ROLE_KEY` (server-side only, not prefixed with NEXT_PUBLIC_)
7. Auto-deploy on git push to `main`

## Success Criteria

- Two browser windows open on the same URL show identical map state
- Assigning a unit in one window updates the other within 200ms
- Data persists after page refresh
- Login works with Supabase credentials
- App is accessible at a public Vercel URL
- All existing features (zones, closures, alerts, radio, demo player) work unchanged
- `/publico` remains accessible without login
- Scenario/demo player state is preserved across page refreshes via localStorage
